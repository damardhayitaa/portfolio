01 — Context

I found the problem while trying to solve a different one

It started with a simple task: build a Mixpanel dashboard to track product metrics for a bi-weekly stakeholder update. But the moment I opened Mixpanel and started pulling data, something was clearly off.

Events had three different names for the same action. Some flows had no tracking at all. Others had events firing for UI elements that didn't lead to any meaningful user behavior — just noise cluttering the system. And there was no documentation anywhere explaining what any of it meant.

What I found

The data existed — but it couldn't be trusted. And if I couldn't trust it, neither could the stakeholders I was supposed to be reporting to.

02 — The Three Problems

It wasn't one issue — it was three compounding each other

After auditing the full Mixpanel setup, I categorized the problems clearly before proposing any solution:

Problem 1

No naming convention

The same user action could be tracked as "button_click", "btn_clicked", or "click_cta_button" — depending on who implemented it and when. Marketing couldn't build dashboards because they didn't know which event to use.

Problem 2

Events that didn't earn their place

A significant portion of events tracked pure UI interactions — hover states, passive scrolls, decorative element renders — that generated no actionable insight. They existed only to exist, cluttering every report.

Problem 3

No shared understanding

Even within the product team, different people defined the same metric differently. "Activation" meant one thing to product and another to marketing. There was no documentation to resolve the gap.

Consequence

Nobody trusted the data

Stakeholder updates required manual reconciliation every time. The numbers in one report didn't match the numbers in another. And when numbers don't match, decisions don't get made.

"The data was there. But nobody could read it — and nobody could agree on what it was saying."
03 — What I Did

Fix the foundation first, then build on top of it

I resisted the urge to just build the dashboard and ship it. If the underlying data was wrong, the dashboard would be wrong too. The right order was: clean first, then report.

1

Full Mixpanel audit — catalogued every event

Went through all 200+ events one by one. Tagged each as: keep, rename, deprecate, or investigate. Built a spreadsheet that became the foundation for everything that followed.

2

Created an event taxonomy and naming standard

Introduced an Object_Action format (e.g., "onboarding_step_completed" not "completedStep_onboarding"). Documented the convention, its rationale, and examples in Notion — accessible to everyone.

3

Deprecated events that didn't drive decisions

For every event flagged as "noise," I asked one question: what decision would this data change? If nobody could answer, the event got deprecated. Cleaner data is better than more data.

4

Worked with engineering to re-instrument over 3 sprints

The cleanup required engineering time. I prioritized by impact — high-traffic flows first — and included re-instrumentation tasks in sprint planning. Made the case for why this was worth the investment.

5

Built the bi-weekly stakeholder dashboard

With clean data in place, I built the dashboard I originally set out to make — now with metrics that were accurate, consistently defined, and mapped to decisions stakeholders actually needed to make.

6

Delivered bi-weekly update cadence to key stakeholders

Made the dashboard a regular touchpoint — not a one-time artifact. Sent a bi-weekly summary with metric highlights, trend callouts, and context for any significant changes. Turned data into narrative.

Key decision

I kept the bi-weekly update as a curated summary — not just a link to the dashboard. Raw dashboards require interpretation. A narrative with context gets read and acted on.

04 — Outcomes

What actually changed

Marketing became self-sufficient

With a clear naming convention and documented event taxonomy, marketing could build their own Mixpanel dashboards without asking PM every time.

Feature decisions got data-backed

Roadmap conversations shifted from "I think users do X" to "here's the data — users do X 3× more than Y." Two features were deprioritized based on low adoption data from the clean dashboard.

Stakeholder trust rebuilt

The bi-weekly update became a consistent touchpoint that leadership relied on. Numbers stopped being questioned because they were now consistently sourced and explained.

Documentation that outlasted the work

The Notion event taxonomy became a living document the team still references. New engineers use it when adding tracking. New PMs use it to understand what's already measured.

05 — What I Learned

Lessons I'm taking forward

01

If you can't trust the data, don't skip the cleanup

It would have been faster to just build the dashboard and present it. But a dashboard built on bad data is worse than no dashboard — it creates false confidence. The slower path was the right one.

02

Every event should answer a question

I now apply a simple test to every tracking request: "What decision does this data change?" If nobody can answer, the event doesn't ship. This one principle cut our noise by 30%.

03

Stakeholder communication is a product too

The bi-weekly update wasn't just a report — it was a product with users, feedback loops, and an outcome to optimize for. Treating it that way (clear structure, consistent cadence, narrative framing) made it actually useful.
